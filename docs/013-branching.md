# Branching Strategy

This document defines how branches, pull requests, and releases are managed in the `gt` repository. It follows a simplified **GitHub Flow** model, appropriate for a single-maintainer CLI project.

## Core Principles

1. **`main` is always releasable.** Any commit on `main` should build, pass tests, and be safe to ship.
2. **All changes go through pull requests.** No direct commits to `main`.
3. **Short-lived branches.** Feature branches should merge within days, not weeks. Long-running branches accumulate drift.
4. **Squash merges.** Keep `main` history linear and readable. One logical change = one commit on `main`.
5. **Tags mark releases.** Versioning uses [Semantic Versioning](https://semver.org/) (`vMAJOR.MINOR.PATCH`).

## Branch Types

| Type | Purpose | Base | Lifetime |
|------|---------|------|----------|
| `main` | Production-ready code | — | Permanent |
| `feat/*` | New features or enhancements | `main` | Until merged |
| `fix/*` | Bug fixes | `main` | Until merged |
| `chore/*` | Build, tooling, dependencies, refactors | `main` | Until merged |
| `docs/*` | Documentation-only changes | `main` | Until merged |
| `hotfix/*` | Urgent production fixes on a release tag | latest release tag | Until merged |

There is **no `develop` branch** and **no long-running `release/*` branches**. If you need staging, cut a release candidate tag (`v0.2.0-rc.1`) from `main` rather than maintaining a parallel branch.

## Branch Naming

Format: `<type>/<short-kebab-case-description>`

**Good:**
```
feat/config-id-passthrough
fix/conditional-trailing-slash
chore/update-clap-v4-5
docs/branching-strategy
hotfix/ssh-config-corruption
```

**Avoid:**
```
feature-branch                 # no type prefix
feat/fix_the_thing             # snake_case
feat/MattsWork                 # no topic, uses author name
feat/issue-42                  # reference issue in the PR, not the branch
```

Keep the description under ~40 characters. If the topic doesn't fit, it's too broad — split the work.

## Workflow

### 1. Start from a fresh `main`

```bash
git checkout main
git pull --ff-only
```

### 2. Create a topic branch

```bash
git checkout -b feat/config-id-passthrough
```

### 3. Commit in small, logical steps

```bash
git add src/cli/args.rs
git commit -m "Restructure ConfigCommands to allow external subcommands"
```

See [Commit Messages](#commit-messages) for the expected style.

### 4. Keep the branch current

Rebase (don't merge) from `main` if the branch falls behind:

```bash
git fetch origin
git rebase origin/main
```

Never rebase a branch someone else is collaborating on without coordinating first. For solo branches, rebase freely.

### 5. Open a pull request

Push the branch and open a PR against `main`:

```bash
git push -u origin feat/config-id-passthrough
gh pr create --fill
```

### 6. Address review feedback

Push additional commits. Do **not** squash locally during review — reviewers need to see what changed between iterations. The squash happens on merge.

### 7. Merge and delete

Once approved and CI is green, merge via **squash merge** (see [Merging](#merging)). Delete the branch after merge.

## Commit Messages

### Format

```
<Subject line in imperative mood, <=72 chars>

<Optional body: what changed and why, wrapped at 72 chars>

<Optional trailers: Refs: #123, Fixes: #456>
```

### Rules

- **Imperative mood**: "Add feature" not "Added feature" or "Adds feature".
- **Capitalize the subject**, no trailing period.
- **Body explains *why*, not *what***. The diff shows what changed.
- **Reference issues** in trailers or the body (`Fixes #42`), not in the subject.
- **No AI/agent attribution.** Do not include `Co-Authored-By: Claude …`, "Generated with …", or similar trailers. Commits should read as human-authored.

### Examples

Good:
```
Add passthrough for unknown git config subcommands

Unknown subcommands under `gt config` now forward to `git config`
verbatim, preserving exit codes and stderr. Explicit gt subcommands
(`id`, etc.) continue to be handled internally.

Fixes #12
```

Avoid:
```
updated the config command                                    # lowercase, past tense, vague
Fixes the bug                                                 # what bug?
Add passthrough (auto-generated by Claude Code)               # no AI attribution
```

## Pull Requests

### Title

Match the commit subject: imperative mood, under 72 chars, no issue numbers.

### Description template

```markdown
## Summary
<1–3 bullets: what this change does and why>

## Changes
<bulleted list of concrete changes>

## Testing
<what you ran, what passed, how to reproduce>

## Related
Fixes #123
Refs #456
```

### Rules

- **Link the issue.** Every PR should reference at least one issue unless it's a trivial typo fix.
- **One logical change per PR.** If the PR covers two unrelated concerns, split it.
- **Keep PRs small.** PRs over ~400 lines of non-generated diff are harder to review well. Split large work into a sequence of dependent PRs.
- **Draft PRs are welcome** for work-in-progress you want early feedback on.
- **No AI/agent attribution** in the PR body or comments. Write as a human.

### Before requesting review

- [ ] `cargo build` succeeds
- [ ] `cargo test` passes
- [ ] `cargo clippy -- -D warnings` is clean
- [ ] `cargo fmt --check` passes
- [ ] Docs updated if behavior changed
- [ ] `CHANGELOG.md` entry added under `## [Unreleased]` if user-visible (see [014-releases.md](014-releases.md#changelog-workflow))

## Merging

### Default: Squash merge

All PRs merge into `main` as a single squashed commit. The squash commit message uses the PR title + description.

### When to rebase merge

Reserved for exceptional cases where preserving the individual commits matters (e.g., a multi-commit refactor where each commit is a distinct, reviewable unit). Coordinate with the maintainer before choosing this path.

### When to merge commit

Never. We do not use merge commits on `main`.

### Branch deletion

Delete feature branches immediately after merge. Stale branches clutter the remote and invite confusion.

## Releases

The full release procedure (CHANGELOG workflow, version bump, release-notes template, checklist, `cargo-release` adoption plan) lives in [014-releases.md](014-releases.md). The short version follows.

### Versioning

`gt` uses Semantic Versioning. Until the first stable release, all versions are `0.x.y`:

- Breaking changes to CLI syntax, config format, or on-disk output (pre-1.0): `MINOR` bump. Reserved for post-1.0: `MAJOR` bump.
- New features, backward-compatible: `MINOR` bump.
- Bug fixes, backward-compatible: `PATCH` bump.

See the [version-bump table in 014-releases.md](014-releases.md#version-bump) for the full matrix.

### Cutting a release (short form)

1. Ensure `main` builds, tests, lints, and formats cleanly.
2. Open a `release/vX.Y.Z` branch. Bump `Cargo.toml`, move `CHANGELOG.md` `[Unreleased]` entries into a dated `[X.Y.Z]` section, update link references at the bottom of the CHANGELOG.
3. Open a PR titled `Release vX.Y.Z`. Squash-merge once CI passes.
4. Tag `main`:
   ```bash
   git checkout main
   git pull --ff-only
   git tag -a vX.Y.Z -m "Release vX.Y.Z"
   git push origin vX.Y.Z
   ```
5. Create a GitHub Release from the tag using the [Release Notes Template](014-releases.md#release-notes-template).

### Release candidates

For pre-release testing, tag `vX.Y.Z-rc.1`, `vX.Y.Z-rc.2`, etc. from `main`. Do not create a `release/*` branch for RCs.

## Hotfixes

If a critical bug ships in a released version:

1. Create `hotfix/<description>` **from the release tag**, not from `main`:
   ```bash
   git checkout -b hotfix/ssh-config-corruption v0.2.0
   ```
2. Fix the bug with minimal scope. No refactors, no unrelated changes.
3. Open a PR against `main`. Merge via squash as normal.
4. Tag a patch release (`v0.2.1`) from `main` after merge.
5. If the hotfix does not apply cleanly to `main` (because `main` has moved on), adapt and land it on `main` separately — the hotfix branch serves the patch release, the equivalent fix on `main` serves the next minor release.

## Quick Reference

| Action | Command |
|--------|---------|
| Start feature | `git checkout -b feat/my-topic main` |
| Update from main | `git fetch origin && git rebase origin/main` |
| Open PR | `gh pr create --fill` |
| View PR checks | `gh pr checks` |
| Merge (squash) | `gh pr merge --squash --delete-branch` |
| Tag release | `git tag -a v0.2.0 -m "Release v0.2.0" && git push origin v0.2.0` |
